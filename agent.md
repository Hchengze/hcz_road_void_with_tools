# HCZ_road_void_with_tools Agent Instructions

## 1. 项目定位

本项目面向城市道路地下空洞、脱空、松散区等浅层异常体探测，目标是利用道路附近或道路既有通信光纤，结合人工震源、车辆、锤击、可控震源等激发方式，建立一套面向三维道路场景的算法原型，内容包括：

1. 三维道路场景正演模拟；
2. 多炮、多道记录下的三维定位反演；
3. 定位结果的不确定性分析；
4. 针对单侧光纤、单侧震源、有限观测孔径等实际条件的可识别性分析。

本项目不是简单的二维剖面问题，也不是二维算法的表面三维化。所有核心算法、数据结构、几何关系和结果解释都必须始终围绕三维场景展开。

核心空间坐标约定如下：

* `x`：沿道路方向或沿光纤方向；
* `y`：横穿道路方向；
* `z` 或 `depth`：深度方向，向下为正；
* `source_xyz`：震源三维坐标；
* `receiver_xyz`：接收点三维坐标；
* `receiver_polyline`：光纤三维折线轨迹；
* `void_xyz`：异常体中心三维坐标；
* `void_radius / void_size_xyz`：异常体尺度参数。

任何模块不得只使用 `x-depth` 二维剖面来替代真实三维问题。二维模型、二维剖面和二维可视化只能作为教学、调试或降维对比工具，不能作为项目主线结果。

## 2. 项目目录说明

当前项目根目录为：

```text
HCZ_road_void_with_tools/
├── code/
├── reference/
└── tools/
```

各目录职责如下：

### 2.1 code

`code/` 是本项目主要代码目录。后续开发的 Python 包、示例、测试、文档、配置文件应优先放在该目录下。

推荐结构如下：

```text
code/
├── hcz_road_void/
│   ├── __init__.py
│   ├── geometry/
│   ├── models/
│   ├── forward/
│   ├── receivers/
│   ├── localization/
│   ├── uncertainty/
│   ├── visualization/
│   └── io/
├── examples/
├── tests/
├── docs/
└── pyproject.toml
```

如现有结构不同，应先检查已有内容，不得盲目重建或破坏已有文件。

### 2.2 reference

`reference/` 存放与绕射波定位、道路空洞探测、浅层异常体识别、DAS/光纤观测、地震波传播等相关文献和参考资料。

Codex 需要优先阅读并整理其中与以下内容有关的方法：

1. 绕射波识别；
2. 空洞或异常体定位；
3. 多炮、多接收点定位；
4. 走时曲线、双曲线、绕射响应、能量聚焦、相干扫描；
5. 空洞深度、横向位置和沿线位置的可识别性；
6. 单侧观测条件下的多解性与不确定性。

文献方法应转化为本项目可执行的算法设计，不能只做摘要。引用文献思想时，应在 `code/docs/REFERENCE_METHODS.md` 中记录来源、适用条件、限制和本项目采用方式。

### 2.3 tools

`tools/` 存放外部开源代码、开源算法包或参考实现。Codex 必须仔细阅读和理解该目录中的代码，建立本项目内部的工具知识库，然后再决定如何复用。

复用方式包括：

1. 直接调用外部工具的公开 API；
2. 参考其算法思想后重新实现轻量版本；
3. 复制小段通用代码并标明来源和许可证；
4. 改写为适合本项目三维道路场景的数据结构；
5. 仅作为验证或对比工具，不进入主流程。

不得盲目整包复制外部代码。不得把外部工具的复杂工程结构强行搬进本项目。不得在未检查许可证的情况下复制源代码。

## 3. 开发总原则

### 3.1 三维优先原则

本项目所有核心模块必须以三维为主线设计。最低要求如下：

1. 几何对象必须使用三维坐标；
2. 震源必须使用 `source_xyz`；
3. 接收点或光纤必须使用 `receiver_xyz` 或 `receiver_polyline`；
4. 异常体位置必须使用 `void_xyz`；
5. 定位搜索空间必须至少包含 `x, y, depth` 三个方向；
6. 结果输出必须说明异常体的沿路位置、横向位置和深度；
7. 不确定性分析必须讨论横向位置和深度之间的混淆风险。

任何只返回二维 `x-depth` 结果的核心算法，都必须标注为临时简化版本，并提供后续三维扩展计划。

### 3.2 物理一致性原则

几何、正演、接收、定位和可视化模块必须共享同一套物理参数和坐标系统。不得出现以下问题：

1. geometry 使用一套异常体参数，forward 使用另一套参数；
2. scan 使用的速度模型与 forward 不一致；
3. receiver 坐标和可视化坐标不一致；
4. depth 方向正负号混乱；
5. source、receiver、void 的单位不一致；
6. 正演结果声称为三维，但实际只计算二维剖面；
7. 定位反演使用的观测量与正演输出物理量不一致。

必须优先建立统一配置对象，例如：

```python
RoadVoidScenario
VelocityModel3D
SourceArray3D
ReceiverGeometry3D
VoidModel3D
ForwardConfig
LocalizationConfig
```

所有模块应通过这些对象传递参数，避免在不同脚本中重复硬编码。

### 3.3 最小可用原型原则

本项目第一阶段目标不是一次性做成工业级软件，而是建立可信、可测试、可解释的三维算法原型。

优先级如下：

1. 先保证三维几何、参数和数据流正确；
2. 再保证正演模拟结果物理含义清楚；
3. 再实现定位扫描和不确定性分析；
4. 最后再考虑高性能、复杂边界、GPU、真实 DAS 数据接入和图形界面。

不得为了追求“看起来高级”而引入过多未验证模块。

### 3.4 复用开源工具原则

`tools/` 中的开源代码应作为重要参考，但复用必须经过审计。

Codex 在使用任何外部工具前必须完成：

1. 识别工具名称、用途、语言、入口文件；
2. 判断是否支持三维；
3. 判断是声波、弹性波、黏弹性、谱元、有限差分还是其他方法；
4. 判断输入输出数据结构；
5. 判断是否适合道路空洞、浅层异常体、DAS/光纤观测；
6. 检查许可证；
7. 判断复用方式：直接调用、轻量改写、只参考算法、暂不使用。

审计结果写入：

```text
code/docs/TOOLS_INVENTORY.md
```

必须避免“没有读懂工具就直接粘贴代码”的行为。

### 3.5 可测试原则

每个核心模块都必须有测试。最低测试要求：

1. 三维坐标维度测试；
2. 单位和坐标方向测试；
3. source、receiver、void 参数同步测试；
4. 速度模型形状测试；
5. 正演输出维度测试；
6. 定位搜索网格维度测试；
7. 简单合成异常体定位回收测试；
8. 异常参数输入测试，例如负速度、负深度、空数组、维度不匹配。

不得只写示例而不写测试。

## 5. 正演模拟设计要求

### 5.1 最低物理要求

正演模块至少应能表达：

1. 三维震源位置；
2. 三维接收位置或光纤轨迹；
3. 三维异常体位置；
4. 背景速度；
5. 异常体速度扰动；
6. 震源子波；
7. 多炮记录；
8. 多道记录；
9. 绕射或散射响应；
10. 输出 `time × receiver × source` 或等价数据结构。

### 5.2 推荐数据结构

建议正演输出对象包含：

```python
ForwardResult3D:
    data
    time_axis
    source_xyz
    receiver_xyz
    receiver_polyline
    model
    void_model
    config
    metadata
```

其中 `metadata` 必须记录：

1. 坐标单位；
2. depth 正方向；
3. 速度单位；
4. 采样率；
5. 子波参数；
6. 正演类型；
7. 是否为完整波动方程；
8. 是否为运动学近似；
9. 是否使用外部工具；
10. 外部工具来源。

### 5.3 DAS/光纤观测要求

本项目最终接收对象不是普通检波器，而是道路附近或既有通信光纤。因此接收模块必须预留 DAS 观测接口。

第一阶段可以先实现点接收器，但应预留：

1. 光纤折线 `receiver_polyline`；
2. 光纤采样点；
3. 光纤方向向量；
4. gauge length；
5. 沿光纤方向应变或应变率观测算子；
6. 点接收记录到 DAS 近似记录的转换接口。

如果第一阶段暂不实现真实 DAS 应变观测，必须在文档中明确说明。

## 6. 定位反演设计要求

定位反演模块后续应参考 `reference/` 中的绕射波定位方法，但必须改写为三维。

最低目标是建立三维搜索框架：

```python
search_x
search_y
search_depth
```

输入应包括：

1. 多炮记录；
2. 多接收点或光纤记录；
3. source_xyz；
4. receiver_xyz / receiver_polyline；
5. 背景速度或速度模型；
6. 候选异常体搜索范围；
7. 目标函数类型。

输出应包括：

1. 最优异常体位置；
2. 三维目标函数体；
3. 沿 x-y、x-depth、y-depth 的切片；
4. 定位置信度；
5. 多解性指标；
6. 横向位置与深度混淆分析。

不得只输出一个最大值点而不分析不确定性。

## 7. 不确定性分析设计要求

不确定性分析是本项目核心内容之一，不能作为附属图件处理。

后续应至少考虑：

1. 噪声扰动对定位结果的影响；
2. 速度模型误差对深度定位的影响；
3. 单侧光纤导致的横向不确定性；
4. 单侧震源导致的照明不均；
5. 异常体尺度与埋深之间的耦合；
6. 多个异常体或干扰体造成的多解性；
7. 不同震源位置组合对定位稳定性的影响。

第一阶段可以先设计接口和文档，后续逐步实现。

## 8. 编程规范

### 8.1 语言和风格

主代码优先使用 Python。要求：

1. 函数和类命名清晰；
2. 类型注解尽量完整；
3. 核心算法使用中文教学型注释；
4. 简单胶水代码可少注释；
5. 不写无意义的大段 AI 式注释；
6. 不使用全局变量传递核心参数；
7. 不在示例脚本中隐藏核心算法。

### 8.2 错误处理

1. 坐标维度不是三维；
2. 速度小于等于零；
3. 采样率小于等于零；
4. 时间轴为空；
5. source 或 receiver 数组为空；
6. depth 方向不一致；
7. 异常体尺寸为负；
8. 搜索范围为空；
9. 外部工具不存在或不可导入；
10. 外部工具输出维度不符合预期。

错误信息应清楚说明问题原因和修正建议。

### 8.3 示例要求

每个示例必须说明：

1. 场景含义；
2. 道路方向；
3. 光纤位置；
4. 震源位置；
5. 异常体真实位置；
6. 正演类型；
7. 输出图件含义；
8. 当前示例的简化条件。

示例不得只生成图片而不解释物理含义。

### 8.4 测试要求

测试应覆盖：

1. geometry；
2. velocity model；
3. void model；
4. forward config；
5. receiver geometry；
6. forward result shape；
7. localization search grid；
8. parameter consistency；
9. invalid input；
10. example smoke test。

## 13. 长期路线

建议长期路线分为以下阶段：

### Stage 1：三维几何与正演最小原型

建立统一三维场景、速度模型、异常体模型、震源接收几何和最小正演输出。

### Stage 2：三维绕射定位反演

参考 reference 中的方法，实现三维走时扫描、能量聚焦、相干扫描或匹配滤波定位。

### Stage 3：DAS/光纤观测算子

将点接收记录扩展为沿光纤方向的应变或应变率观测，考虑 gauge length 和光纤走向。

### Stage 4：不确定性分析

分析速度误差、噪声、单侧光纤、单侧震源、有限孔径和异常体尺度对定位结果的影响。

### Stage 5：高保真正演工具接入

根据 tools 审计结果，逐步接入 Devito、SPECFEM3D、OpenSWPC、SeisCL 或其他三维正演工具。

### Stage 6：真实数据适配

接入实际 DAS 数据、锤击记录、车辆激发记录或可控震源记录，建立预处理、事件检测、定位和解释流程。
